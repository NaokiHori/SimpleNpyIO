Implementation - writer¶

Write header of npy file.

Definition¶

size_t simple_npyio_w_header(
  const size_t ndim,
  const size_t *shape,
  const char dtype[],
  const bool is_fortran_order,
  FILE *fp
);

size of header which is written

Warning

shape should have ndim elements. Allocating its memory correctly is the users’ responsibility to avoid undefined behaviours.

Note

File should be opened a priori by the user with a writable mode (e.g. "w").

Note

This function returns header_size, which is the size of header written to the given file stream fp when all succeeded. 0 is returned, on the other hand, when failed. It is recommended to check the returned value.

Description¶

Wiring procedure of a npy header is taken care of by a main function simple_npyio_w_header, which consists of the following process:

Detailed descriptions can be found hereafter.

Note

All snippets come from src/simple_npyio.c.

  1. Prepare magic string

    All npy files start with a fixed magic characters \x93NUMPY, which is defined as a global variable at the beginning of the source.

    15static const char magic_string[] = {"\x93NUMPY"};

  2. Decide minor version

    For the time being minor version is fixed to 0.

    1243  const uint8_t minor_version = 0;

  3. Create dictionary

    Dictionary contains three components, data type, memory order, and shape of the dataset, which are taken care of by a function create_dict.

    1245  char *dict = NULL;
1246  size_t n_dict;
1247  if(create_dict(&dict, &n_dict, ndim, shape, dtype, is_fortran_order) != 0){
1248    error_handlings();
1249    return 0;
1250  }

    Note

    As can be seen, this function returns 0, indicating the function failed. The total size of the created dictionary n_dict is returned on the other hand, when all procedures succeed. This behaviour is consistent for all functions below.

    More details: create_dict

    Three keys are needed to be a correct npy file, whose values and corresponding dictionary are generated in this function.

    • data type of the dataset

      More details

      Key is 'descr', whose value is given by create_descr_value. For now user input is simply copied and used as a data type, and as a result users are responsible for giving a correct data type. For proper datatypes which can be interpreted by Numpy, please refer to the documentation.

      1037  if(create_descr_value(&descr_value, dtype) != 0){
1038    ERROR("create_descr_value failed\n");
1039    return -1;
1040  }

      Note that -1 is used as a returned value representing failure of a function, while 0 is used to say the function is terminated successfully.

    • memory order of the dataset

      More details

      Key is 'fortran_order', whose value is given by create_fortran_order_value. This function takes a boolean input is_fortran_order, which is converted to a corresponding string True or False. Since this library is written in C, the value is False normally.

      1042  if(create_fortran_order_value(&fortran_order_value, is_fortran_order) != 0){
1043    ERROR("create_fortran_order_value failed\n");
1044    return -1;
1045  }

    • data sizes of the dataset

      More details

      Key is 'shape', whose value is given by create_shape_value. It should be a Python tuple, e.g.,

      value (char *)

      description

      ()

      scalar

      (3,)

      one-dimensional array (vector) whose size is 3

      (3,5,)

      two-dimensional array whose outer and inner sizes are 3 and 5, respectively

      and so on

      This function generates this kind of data from the given ndim and shape.

      1047  if(create_shape_value(&shape_value, ndim, shape) != 0){
1048    ERROR("create_shape_value failed\n");
1049    return -1;
1050  }

    These results are assigned to a variable elements

    1052  const size_t n_elements_dict = 13;
1053  char **elements = smt_calloc(n_elements_dict, sizeof(char *));
1054  // initial wave bracket
1055  elements[ 0] = "{";
1056  // 'descr':descr_value
1057  elements[ 1] = (char *)descr_key;
1058  elements[ 2] = ":";
1059  elements[ 3] = (char *)descr_value;
1060  elements[ 4] = ",";
1061  // 'fortran_order':fortran_order_value
1062  elements[ 5] = (char *)fortran_order_key;
1063  elements[ 6] = ":";
1064  elements[ 7] = (char *)fortran_order_value;
1065  elements[ 8] = ",";
1066  // 'shape':shape_value
1067  elements[ 9] = (char *)shape_key;
1068  elements[10] = ":";
1069  elements[11] = (char *)shape_value;
1070  // final wave bracket
1071  elements[12] = "}";

    to build up the final dictionary (char *dict) such as

    {'descr':'<i4','fortran_order':False,'shape':(3,5,)}

    Note

    Arbitray number of spaces (outside quotations since keys should be as they are) can be added to the dictionary (as we normally do to improve visibility). Also pair of single quotations can be replaced by pair of double quotations. These differences are trivial for writing process, while the reader should take care of them.

  4. Decide major version and padding

    Major version depends on the size of the dictionary, which is the reason why deciding it is postponed.

    The total size of the header excluding metadata (magic_string and versions) needs to be stored in HEADER_LEN later, whose data type is unsigned short (normally 2 bytes) by default (major version is 1). Thus, when HEADER_LEN exceeds 65535 (USHRT_MAX: maximum value for an unsigned short value to hold), it fails to keep the correct data size. In order to avoid this issue, major version of 2 must be used for datasets whose header size is moderately large, when the datatype of HEADER_LEN is upgraded to unsigned int (normally 4 bytes).

    Also total header size should be divisible by 64; in order to achieve this, some paddings are added at the end of the header.

    1252  uint8_t major_version;
1253  uint8_t *padding = NULL;
1254  size_t n_padding;
1255  if(create_padding(&padding, &n_padding, &major_version, n_dict) != 0){
1256    error_handlings();
1257    return 0;
1258  }
    More details: create_padding

    Sizes of elements which are already known are computed first:

    1124  size_t n_magic_string = strlen(magic_string);
1125  size_t size_magic_string  = sizeof(char)*n_magic_string;
1126  size_t size_major_version = sizeof(uint8_t);
1127  size_t size_minor_version = sizeof(uint8_t);
1128  size_t size_dict          = sizeof(char)*n_dict;

    Even when the major version is 2, header size larger than UINT_MAX cannot be stored, which is checked here:

    1130  if(size_dict > UINT_MAX-64){
1131    ERROR("size of dictionary is huge (%zu)\n", size_dict);
1132    return -1;
1133  }

    Similarly, USHRT_MAX (a bit looser for safety) is used as a threshold to decide the major version as well as size_header_len (datasize of HEADER_LEN)

    1135  // large portion of the header is occupied by dict
1136  // so check dict size, and if it is larger than USHRT_MAX-64,
1137  //   use major_version = 2
1138  size_t size_header_len;
1139  if(size_dict > USHRT_MAX-64){
1140    *major_version = 2;
1141    size_header_len = sizeof(uint32_t);
1142  }else{
1143    *major_version = 1;
1144    size_header_len = sizeof(uint16_t);
1145  }

    Now we know the data sizes of all elements in the header except the padding:

    1147  size_t size_except_padding =
1148    +size_magic_string
1149    +size_major_version
1150    +size_minor_version
1151    +size_header_len
1152    +size_dict;

    Padding exists to make the total header size divisible by 64, which is achieved by

    1154  // increase total size by 64 until becoming larger than size_except_padding
1155  // NOTE: size_padding == 0 is NOT allowed since '\n' is necessary at the end
1156  //   thus the condition to continue loop is "<=", not "<"
1157  size_t size_header = 0;
1158  while(size_header <= size_except_padding){
1159    size_header += 64;
1160  }
1161  size_t size_padding = size_header-size_except_padding;

    Finally the padding is created, which consists of many spaces 0x20 and last single newline character 0x0a:

    1163  *n_padding = size_padding/sizeof(uint8_t);
1164  *padding = smt_calloc(*n_padding, sizeof(uint8_t));
1165  // many ' 's: 0x20
1166  memset(*padding, 0x20, sizeof(uint8_t)*(*n_padding-1));
1167  // last '\n': 0x0a
1168  (*padding)[*n_padding-1] = 0x0a;

  5. Compute HEADER_LEN

    HEADER_LEN is well explained in the documentation of the npy format. Roughly speaking, it is the total size of dictionary and padding. This number, however, needs to be written in little-endian form, and can have different data size depending on the major version.

    1260  uint8_t *header_len = NULL;
1261  size_t n_header_len;
1262  if(create_header_len(&header_len, &n_header_len, major_version, n_dict, n_padding) != 0){
1263    error_handlings();
1264    return 0;
1265  }
    More details: create_header_len

    Dictionaries having too large size (and padding just in case) are rejected for safety:

    1180  // Here "too large" means header size (not data size)
1181  //   is larger than approx. 2GB, which would not happen normally
1182  if(n_dict >= UINT_MAX/2){
1183    ERROR("dictionary size is huge (%zu)\n", n_dict);
1184    return -1;
1185  }
1186  if(n_padding >= UINT_MAX/2){
1187    ERROR("padding size is huge (%zu)\n", n_padding);
1188    return -1;
1189  }

    Note that this is not talking about the size of data, but size of header.

    HEADER_LEN is computed here:

    1191  if(major_version == 2){
1192    // major version 2, use uint32_t to store header_len
1193    uint32_t header_len_uint32_t = (uint32_t)n_dict+(uint32_t)n_padding;
1194    *n_header_len = sizeof(uint32_t)/sizeof(uint8_t);
1195    *header_len = smt_calloc(*n_header_len, sizeof(uint8_t));
1196    memcpy(*header_len, &header_len_uint32_t, *n_header_len);
1197    LOGGING("header_len (uint32_t): %u\n", header_len_uint32_t);
1198  }else{
1199    // major version 1, use uint16_t to store header_len
1200    uint16_t header_len_uint16_t = (uint16_t)n_dict+(uint16_t)n_padding;
1201    *n_header_len = sizeof(uint16_t)/sizeof(uint8_t);
1202    *header_len = smt_calloc(*n_header_len, sizeof(uint8_t));
1203    memcpy(*header_len, &header_len_uint16_t, *n_header_len);
1204    LOGGING("header_len (uint16_t): %hu\n", header_len_uint16_t);
1205  }

    It is first computed naively, e.g., for major_version = 1,

    uint16_t header_len_uint16_t = (uint16_t)n_dict+(uint16_t)n_padding;

    which is stored as an array of uint8_t, e.g., for major_version = 1,

    *n_header_len = sizeof(uint16_t)/sizeof(uint8_t);
*header_len = my_calloc(*n_header_len, sizeof(uint8_t));
memcpy(*header_len, &header_len_uint16_t, *n_header_len);

    where a buffer header_len, whose data type is uint8_t and number of items is n_header_len, is allocated and assigned.

    Finally the endian of this array is converted if the architecture is big-endian:

    1207  if(is_big_endian()){
1208    if(convert_endian(header_len, sizeof(*header_len)) != 0){
1209      ERROR("convert_endian failed\n");
1210      return -1;
1211    }
1212  }
